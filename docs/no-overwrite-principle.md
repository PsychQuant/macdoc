# No-Overwrite Principle

## 核心原則

**AI 額外產生或修正的內容，除非有明確必要，絕不覆蓋。**

所有 macdoc pipeline 中的自動化步驟（normalize、fix-envs、consolidate 等）
必須遵守此原則，確保已完成的工作不會因重複執行而遺失。

## 實作要求

### 1. 冪等性（Idempotency）

每個機械修正步驟都必須是冪等的——跑一次和跑十次，結果相同：

| 步驟 | 冪等機制 |
|------|---------|
| Document class 修正 | 只在 `{article}` 存在時才替換為 `{book}` |
| 數學運算子偵測 | 檢查 preamble 是否已有 `\DeclareMathOperator` 定義 |
| 貨幣 `$` 跳脫 | `(?<!\\)` lookbehind 跳過已跳脫的 `\$` |
| 符號替換 | 替換已完成的字串不會再匹配 |
| 跨頁重複移除 | 已移除的行不會再匹配 |
| 環境修復 | `check()` 回傳空陣列時 `fix()` 不執行 |

### 2. 備份保護

- 首次執行 `consolidate` 時建立 `.bak` 備份
- **已存在的 `.bak` 不會被覆蓋**，保護最初的原始版本
- 如需強制重新備份，使用者必須手動刪除 `.bak`

### 3. 分層修改

```
Layer 1: 機械修正（normalizer, env-checker）
  → 確定性操作，可重複執行，結果一致

Layer 2: AI agent 修正（consolidator agent loop）
  → 非確定性操作，agent 的修改在下次 Layer 1 跑時不應被覆蓋

Layer 3: 人工修正
  → 最高優先，任何自動化步驟都不應覆蓋人工編輯
```

### 4. 多檔案協調

- `normalizeProject` 同時處理主檔和外部 preamble
- 兩個檔案的修改是原子性的：先完成所有分析，最後才寫回
- 只寫回確實有變動的檔案

## 開發者注意事項

撰寫新的自動修正邏輯時：

1. **先檢查再修正**：在套用修正前，確認目標狀態尚未達成
2. **不假設初始狀態**：程式可能在任何階段的中間狀態被呼叫
3. **保留語意**：修正格式問題時，不改變內容語意
4. **記錄變更**：透過 report struct 回報每個步驟的實際變動數量
5. **測試冪等性**：每個新功能都必須有「連續跑兩次，第二次無變動」的測試
